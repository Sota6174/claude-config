# Global CLAUDE.md (user-level)

このファイルは全プロジェクト共通の Claude Code 規約を置く。プロジェクト固有のルールは各ディレクトリの CLAUDE.md 側に書く。

## 複雑タスク受領時の事前アライン規約

依頼を受けた直後に、それが下記トリガーのいずれかに該当すると判断したら、**コード/ファイル変更・調査スパウンに着手する前に一度だけ**、`AskUserQuestion` ツールで方針確認を取る。

### トリガー条件（いずれかに該当したら発火）

1. **複数ファイルにまたがる実装・リファクタ** — 2 ファイル以上の編集または新規作成が見込まれる、または既存ディレクトリ構造・スクリプトの枠組みに手を入れる
2. **設計・方針の決定を伴うもの** — 「どう作るか」「どちらを選ぶか」の選択肢が複数あり、後戻りコストが小さくない（採用したライブラリ、データスキーマ、ファイル配置、コマンド体系等）
3. **前提条件・要件が曖昧な依頼** — 入力/出力、スコープ、成果物の形式、成功条件のうちいずれかが言語化されていない、または複数解釈が成立する

### 確認の取り方

- **必ず `AskUserQuestion` ツールを使う**（長文の自然文で延々と聞き返さない）
- 本文では 2〜3 文の状況整理だけ出し、すぐツールを呼ぶ
- 質問の中身は次の 3 観点から、その依頼に必要なものだけを選んで合計 1〜3 問にまとめる:
  1. **方針** — 自分が想定したアプローチを 1〜2 行で示し、それで合っているか
  2. **前提** — 自分が暗黙に置いている前提（対象範囲、入力形式、既存ファイルの扱い、上書き可否等）を明示し、認識ずれがないか
  3. **質問** — 依頼文から読み取れず判断保留している点
- **1 ラウンドで終わらせる**。回答が来たら追加質問せずに着手する

### やってはいけないこと

- **トリガーに当たらない依頼で確認する** — 単一ファイル修正、事実調査、用語説明、Markdown 数行追記、軽微なバグ修正、明確に指示された一発作業
- ユーザーが「お任せ」「いいから進めて」「とりあえずやってみて」「ドラフトだけ作って」と明示している依頼で確認を挟む — その場合は判断ベースで着手し、気づきがあれば作業中に短く報告する
- 着手後に「やっぱり方針確認します」と途中で止まる — 判定はあくまで着手前の 1 回限り
- 質問を 4 個以上並べる、長文の前置きを置く、回答に圧をかけるような誘導的選択肢にする

### 迷ったときの既定

トリガー該当か自信がない場合は **確認しない側に倒す**（過剰確認は作業効率と学習リズムを阻害するため）。ただし「後戻りコストが高そう」(設定ファイル/永続データ/外部公開物の生成・変更、長文ドキュメントの全面書き換え等) と感じたら確認する側に倒す。

## シェル / コマンド提案の規約

ユーザーの実行環境は**基本的に bash**（Windows 上でも Git Bash / WSL / MSYS 等の bash）。コマンドを提案・記載するときは次に従う。

- **既定は bash 形式**で提案する。複数行に折り返す場合の継続文字は **`\`（バックスラッシュ）**を使い、`\` の直後は改行（末尾に空白を残さない）。
- **PowerShell でしか実現できない／PowerShell を使う必要があるとき**だけ PowerShell 形式を出す。その際は **「PowerShell 用」であることを明記**した上で、PowerShell の継続文字 **`` ` ``（バッククォート）** を使う。
- bash と PowerShell を混在させて提案しない。どちらの形式かが一目で分かるように、コードブロックの言語指定（` ```bash ` / ` ```powershell `）と見出し・注記を一致させる。
- 参考: 継続文字はシェルごとに異なる — **bash = `\`** / **PowerShell = `` ` ``** / **cmd.exe = `^`**。bash でバッククォートは継続文字ではなくコマンド置換なので使わない。

## ツール運用の落とし穴

### `gh issue view` で issue 本文を読むとき

- issue **本文**を読むなら `gh issue view <n>`（`--comments` を付けない）。
- `--comments` は **コメント一覧のみ** を表示するフラグ。コメント0件の issue では出力が空（exit 0・stdout/stderr 0 バイト）になる。これは正常動作であり、issue が読めない・コマンド失敗ではない。空出力なら `--comments` を外して取り直す。

### Grep / Glob ツールが無い実行環境がある

- 環境によっては `Grep` / `Glob` ツールが提供されない（ツール一覧にも `ToolSearch` の deferred 一覧にも存在しない）。これは hook によるブロックではなく、その harness 構成の仕様。
- その場合は Bash の `grep` / `rg` か `Explore` サブエージェントで代替する（これが正しいフォールバック）。
- 独立した別事象を因果で結びつけない。「あるコマンドが空出力だった」ことと「Grep が使えない」ことは無関係。一方を他方の原因にせず、それぞれ独立に対処する。
