# 公式ドキュメントの効率的取得

実装/修正方針の根拠は、可能な限り **対象バージョンの公式ドキュメント** に揃える。古いブログや別バージョンの情報を引っ張ってくると、後で全部書き直しになる。

## 取得の3層

```
context7 MCP  ──▶  公式 GitHub (CHANGELOG / Issues / Discussions)  ──▶  公式ブログ・WebSearch
（一次情報・構造化）  （バージョン差分・既知問題）              （補強情報）
```

下の層に降りるのは、上の層で答えが得られなかった時だけ。

---

## 第一手：context7 MCP

### 手順

1. `mcp__plugin_context7_context7__resolve-library-id` を呼んで、対象ライブラリの ID を確定する
2. `mcp__plugin_context7_context7__query-docs` を `topic` 引数つきで呼ぶ

### `topic` の絞り方

```
× topic: "Next.js"            （広すぎる、ドキュメント全部取得しに行く）
○ topic: "App Router data fetching with cache options"
○ topic: "Prisma transaction with retry on conflict"
○ topic: "React Server Components error boundary patterns"
```

**コツ：**
- 「ライブラリ名」ではなく「**やりたいこと**」をトピックにする
- バージョン固有の挙動を聞きたい時は、トピックにバージョンを含める（`React 19 useFormStatus changes` 等）
- 同一ライブラリで複数トピックが必要なら、可能なら1つの広めのトピックにまとめる（呼び出し回数の削減）

### context7 で取れない場合

- そのライブラリが context7 の対象外（ID が resolve できない）
- 取得結果が古いバージョンの内容（極稀）
- → 公式 GitHub への WebFetch にフォールバック

---

## 第二手：公式 GitHub

### CHANGELOG / Release Notes

対象バージョンと現行最新バージョンの **間に挟まる変更** を確認する。これを怠ると、「現バージョンでは廃止された機能を使うコードを書く」事故が起きる。

- `https://github.com/<org>/<repo>/releases` を WebFetch
- 大規模ライブラリは `https://github.com/<org>/<repo>/blob/main/CHANGELOG.md` を直接 fetch
- メジャー間の移行は「migration guide」というキーワードで探す

### Issues / Discussions

該当機能の **既知の落とし穴** を確認する。

- `https://github.com/<org>/<repo>/issues?q=is%3Aissue+<キーワード>`
- Pinned Issues / Discussion Sticky は「FAQ レベルの重要事項」が多い
- 「直近で議論が活発な Issue」は仕様が固まっていない可能性、提案前にチェック

### docs/ ディレクトリ

公式ドキュメントサイトの裏側は GitHub に置かれている場合が多い。リポジトリの `docs/` を直接読むと、サイトより新しいことがある（特に未リリースの v-next）。

## 第三手：WebSearch / WebFetch

公式に書かれていない・分散している情報の補強：

- 「<ライブラリ> <機能> best practice 2026」
- 「<ライブラリ> <ライブラリ別> migration guide」
- 「<エラーメッセージそのもの>」（具体的なエラーの解決策探し）

**ルール：**
- 検索結果の **公開日を必ず確認**。半年以上前はバージョン整合の再チェック必須
- 大手技術メディア（MDN / web.dev / 公式ベンダー） > StackOverflow（高評価回答） > 知名度の高いブログ
- 個人ブログは「裏取り対象」であって、それ自体を根拠にしない

---

## 取得する情報の項目

何のために公式ドキュメントを取りに行くのか、目的別の検索パターン：

### A. API の正しい使い方

- 公式 example コード
- 引数の必須/任意・デフォルト値
- 返り値の型・例外

### B. バージョン固有の注意点

- 該当バージョンで非推奨化された機能（deprecated）
- 該当バージョンで追加された機能（既存コードが古い書き方を使っていないか）
- 次メジャーで破壊的変更予定の機能

### C. 推奨アーキテクチャパターン

- 公式が示す「推奨されるパターン」
- アンチパターンとして公式が明示しているもの
- パフォーマンス / セキュリティ上の注意

### D. 既知の落とし穴

- Issue で繰り返し報告されているハマりポイント
- Discussion で公式が「これは仕様」と回答しているもの

## 取得後の整理

取得した情報は **その場で要約してメモ化** する。次の Step（既存パターンとの差分整理）でそのまま使える形に：

```markdown
## <ライブラリ名> v<X.Y> 該当機能の公式 BP（取得日：2026-MM-DD）

**API：**
- `someApi(arg1, arg2?)` — arg2 はバージョン X.Y で追加（出典：CHANGELOG <URL>）

**推奨パターン：**
- 〜 を使う（公式 docs <URL>）

**注意点：**
- 〜 は次バージョンで非推奨（公式 Issue #1234）
- 〜 は同期/非同期で挙動が変わる（公式 docs <URL>）

**避けるべき書き方：**
- 〜
```

このメモが、後段の「既存 vs 公式 BP」比較表の片側になる。

---

## やってはいけない取得パターン

- ❌ 公式ドキュメントを「とりあえず全部取得」する（context を浪費）
- ❌ 1つの情報源だけで決定的な判断を下す（一次情報で必ず裏取り）
- ❌ 取得した情報の公開日を確認しない（古い情報を採用する事故の主因）
- ❌ メジャーバージョンが違うブログの情報を採用する（実害大）
- ❌ 同じドキュメントを何度も取り直す（一度取得したらメモから引く）
