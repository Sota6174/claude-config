# `docs/best-practices/<tech>.md` のテンプレート

頻繁に参照する技術については、公式ドキュメントの該当部分をプロジェクト内に **「採用バージョンと運用ルール込みで」要約したベストプラクティス md** を作る。これにより、毎回ドキュメントを取りに行くコストが消える。

## 何のために作るか

- **毎回の公式ドキュメント取得を省く** — トークンと時間の節約
- **プロジェクト固有の方針を一箇所に集約** — 「このプロジェクトでは公式の選択肢 A/B のうち A を採用する」のような判断記録
- **バージョン固定** — 採用バージョンを明示することで、将来のアップグレード判断材料になる
- **新規参加時のオンボーディング** — 自分自身が時間を空けて戻ってきた時も含めて、規約を一目で把握できる

## どんな時に作るか（提案判断基準）

次のいずれかに該当する場合に提案する：

1. **同じ技術が今後も繰り返し触られる** ことが見込まれる（プロジェクトの中心的依存）
2. **公式ドキュメントの該当セクションが分散** しており、毎回探すコストが大きい
3. **バージョン固有の非自明な落とし穴**がある
4. **プロジェクト固有の運用ルール**（採用しないパターン / 独自の慣習）と組み合わせて書く価値がある

逆に、**一度使って終わり** の技術や、**公式ドキュメントを1ページ読めば十分** な技術には作らない。

## 作成時のコツ

### 短く保つ

200〜400行を目安に。長くなりすぎたら：
- セクションを別 md に分割（`<tech>-testing.md` 等）
- 一般的な API リファレンスを丸写ししない（公式 docs にリンク）

### 「なぜ」を必ず書く

「採用する」「採用しない」の選択には、**プロジェクト固有の根拠** を明記する：

- ❌ 「採用しない」（理由なし）
- ✅ 「採用しない — このプロジェクトはサーバーレス前提で動かすため、常駐プロセス必要な機能 X は使えない」

理由が書かれていないと、将来「なぜこれを避けていたっけ？」となって規約が空文化する。

### 公式へのリンクを必ず付ける

各セクションに公式ドキュメントへの URL を貼る。md だけで自己完結させようとせず、**「ここを起点に公式に飛べる」** 状態にする。

### 確認日を明示

`最終確認日: YYYY-MM-DD` を冒頭に置く。半年以上更新がなければ「再確認すべき」のシグナルになる。

## 提案フォーマット（ユーザーへ）

実装/修正方針提示の最後にこう書く：

```markdown
## BP md 化の提案

このプロジェクトでは `<tech>` を継続的に使っていく方針なので、今回調査した公式 BP を `docs/best-practices/<tech>.md` にまとめておくと、後の実装で毎回ドキュメントを取り直さずに済みます。

**含める予定の内容（要約）：**
- 採用バージョン：<X.Y.Z>
- 採用するパターン：<n 個、例：データフェッチは A、エラーは B>
- 採用しないパターン：<n 個、例：C は使わない、理由：〜>
- バージョン固有の注意：<n 個>

作成しますか？（はい / 不要 / 簡略版で）
```

---

## 既存の BP md を更新する場合

新しい調査で公式 BP が更新されていたら、既存の `<tech>.md` も更新する：

1. 「最終確認日」を今日の日付に
2. 変更された推奨パターンがあれば該当セクションを差し替え
3. 「更新履歴」に行を追加（日付・変更内容・採用バージョン）
4. **既存コードへの影響範囲をユーザーに報告** — 規約が変わると既存コードも徐々に追随する必要が出る
