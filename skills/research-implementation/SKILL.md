# research-implementation

「既存コード」と「変更後のコード」の間にある"差分の設計"を、**既存スタックの正確な把握 × 公式ドキュメントの一次情報** で支える調査スキル。

実装計画や修正方針が公式の推奨パターンとズレていると、書き終えた後で気付いて全部書き直す羽目になる。だからこそ、**手を動かす前に「既存スタック」と「公式 BP」の両方を押さえる**ことを最優先にする。

---

## いつ使うか（トリガーの再確認）

次のような発言があったら発動する：

- 「〜を実装したい」「機能を追加したい」
- 「〜を修正したい」「バグを直したい」「リファクタしたい」
- 「実装計画書を作って」「仕様書をまとめて」「修正計画を立てて」
- 「<ライブラリ名> の正しい書き方は？」「<フレームワーク> の推奨パターンは？」
- 「ベストプラクティスに沿った形に直したい」
- 「<ライブラリ> の最新バージョンでこの書き方は妥当？」

**逆に使わない場面：** まだコードが存在せず、技術スタック自体を決める段階 → [[research-planning]] の領分。

---

## 調査フロー

順序を守ること。**「既存把握 → 公式照合 → 差分整理 → 方針提示」** の順で進めなければ、的外れな提案になりやすい。

### Step 1. 既存スタックを把握する（コードに触る前）

まず作業対象プロジェクトの **技術スタックとバージョン** を把握する。詳細は [[references/stack-detection.md]]。

主要な確認先：
- パッケージマニフェスト（`package.json`, `requirements.txt`, `pyproject.toml`, `go.mod`, `Cargo.toml`, `pom.xml` 等）
- ロックファイル（実際にインストールされているバージョン）
- 設定ファイル（`tsconfig.json`, `.eslintrc`, `next.config.js`, `pytest.ini` 等）
- README.md / CLAUDE.md / docs/ 配下

**バージョンは必ずメジャー.マイナーまで確認する**。古いメジャーバージョンの公式ドキュメントを取得しても役に立たない（むしろ有害）。

### Step 2. プロジェクト固有の BP md を確認する

`docs/best-practices/<tech>.md` のようなプロジェクト固有のベストプラクティス文書が既にあれば、それを最優先で参照する。これは「過去の自分たちが決めた方針」であり、公式 BP より優先度が高い場面が多い（既存コードとの一貫性のため）。

存在する場合：
- 該当する `<tech>.md` を読み、適用すべき規約を把握する
- 公式 BP と矛盾する場合は、矛盾点をユーザーに報告する

存在しない場合：
- 後段の Step 6 で BP md 化を提案する候補としてメモする

### Step 3. 既存コードの該当箇所を把握する

修正対象の周辺コードを **十分な範囲で** 読み、次を抽出する：

- 既存の実装パターン（命名規則、ファイル構成、層分け、エラーハンドリングの流儀）
- 既に使われている API / ヘルパー / 抽象（再利用すべきもの）
- テストの構造（テストを追加・修正する場合に従うべき形）

**全ファイル読みは避ける**。トリガーとなった機能やバグの周辺、および同じパターンを採用しているはずの類似機能を Grep / Glob で見つけ、抜粋して読む。

**広範囲探索が必要な場合は subagent に委譲する**（本セッションの context を節約するため）：

- **数十ファイルにまたがる "どこで X してる？" 系** → `Explore` (read-only fast) サブエージェントに丸投げ。要約だけ返る
- **アーキテクチャ把握 / 層分け解明が中心** → `feature-dev:code-explorer` agent を呼ぶと、実行パス追跡・依存関係マップまで構造化して返してくれる
- **Step 4 (公式ドキュメント取得) と並列実行可能**：「既存把握」と「公式 BP 取得」は依存がないので同一メッセージ内で並列起動する

### Step 4. 公式ドキュメントの最新情報を取得する

[[references/doc-fetching-strategy.md]] に従い、公式ドキュメントを取得する。**Step 1 で把握したバージョン** に該当する情報を取りに行くこと。

- **第一手は context7 MCP**：`mcp__plugin_context7_context7__resolve-library-id` → `query-docs`（`topic` 引数で範囲を絞る）
- 公式 GitHub の Release Notes / CHANGELOG で **対象バージョンと現行最新の差分** を確認
- 公式 Issue / Discussion で「該当機能の既知の落とし穴・推奨パターンの議論」を確認

**取得する情報の範囲：**
- API の正しい使い方（公式の example コード）
- バージョン固有の非推奨警告（deprecated / breaking change）
- 公式が推奨するアーキテクチャパターン（該当する場合）

### Step 5. 既存パターンと公式 BP の差分を整理する

ここが最も価値が出る部分。次の3つに分類する：

| 分類 | 説明 | 扱い |
|---|---|---|
| **整合** | 既存パターン = 公式 BP | そのまま踏襲 |
| **既存独自** | 公式 BP には記述がないがプロジェクト独自に確立されたパターン | 既存に揃える（理由を確認） |
| **乖離** | 既存パターンが公式 BP と異なる、または明確に非推奨 | 影響範囲を見て判断（後述） |

**乖離があった場合の判断軸：**
- 既存コードが大量にその古いパターンを使っているか？（影響範囲）
- 公式が「セキュリティ上の問題で必須移行」と明記しているか？
- 既存パターンが破壊的変更で動かなくなる予定か？

これらにより、次のいずれかを提案する：
1. **既存に揃える**（影響範囲が大きく緊急性が低い場合）
2. **新規分から公式 BP に揃える**（既存はそのまま、新コードのみ正しい形で書く）
3. **既存も含めて公式 BP に揃える**（緊急性が高い、または既存が小規模）

### Step 6. 実装/修正方針を提示する

次の構造で出す：

```markdown
## 既存スタックの把握

- 言語/ランタイム：<例> Python 3.12 / Node 20
- 主要フレームワーク/ライブラリ（バージョン込み）：
  - <例> Next.js 15.0.3 (App Router)
  - <例> Prisma 5.20.0
- プロジェクト固有 BP（あれば）：`docs/best-practices/<tech>.md`

## 関連する公式 BP / 最新情報

- <ライブラリA> v<X> の該当機能：<推奨パターンの要約>（出典：<公式 URL>）
- <ライブラリB>：<バージョン固有の注意点>

## 既存パターン vs 公式 BP

| 観点 | 既存 | 公式 BP | 差分の扱い |
|---|---|---|---|
| 〜 | 〜 | 〜 | 整合 |
| 〜 | 〜 | 〜 | 乖離（新規分のみ揃える） |

## 実装/修正方針

1. <Step 名 / 何をするか>
   - 変更ファイル：<path>
   - 採用パターン：<公式 BP / 既存独自 / 折衷>
   - 根拠：<出典>

2. ...

## テスト方針
- 〜

## 影響範囲とリスク
- 〜

## BP md 化の提案（後述）
- 〜
```

### Step 7. 頻繁参照分の BP md 化を提案する

調査の過程で「この技術はこのプロジェクトで継続的に参照することになる」と判断したら、**公式ドキュメントの該当部分を `docs/best-practices/<tech>.md` にまとめる提案** を行う。詳細は [[references/bp-md-template.md]]。

**BP md 化を提案する判断基準：**
- 同じ技術（ライブラリ / フレームワーク）が **今後も継続的に触られる** ことが見込まれる
- 公式ドキュメントの該当セクションが分散しており、毎回探すコストが大きい
- バージョン固有の非自明な落とし穴や推奨パターンがある
- プロジェクト固有の運用ルールと組み合わせて記述する価値がある

**提案フォーマット：**
```markdown
## BP md 化の提案

このプロジェクトでは `<tech>` を継続的に使うので、以下を `docs/best-practices/<tech>.md` にまとめておくと、後の実装で毎回ドキュメントを取り直さずに済みます：

- 採用バージョン：<X.Y.Z>
- 中心となる推奨パターン（〜行程度）
- このプロジェクトで採用しないパターン・理由
- バージョンアップ時に確認するもの

作成しますか？ 構造のドラフトはこちらです：
（references/bp-md-template.md の構造を提示）
```

ユーザーが OK したら、**実際に `docs/best-practices/<tech>.md` を生成** する（テンプレートに沿う）。

## トークン効率の原則

[[../research-planning/references/efficient-fetching.md]] に従う。重要ポイント：

- **Step 1〜3 の既存把握は Grep / Glob を駆使して必要箇所だけ読む**。全ファイル読みは禁止
- **context7 の `topic` を必ず指定**して公式ドキュメント取得範囲を絞る
- **同じドキュメントを2度取得しない**。最初の取得時に必要情報を抜粋して使い回す
- **大きなプロジェクトでは並列 Agent (Explore)** で「既存パターンの調査」と「公式ドキュメント取得」を同時実行することを検討

## 最後に：実装に着手する前に必ず方針を提示する

このスキルの出力は **「次に何をどう書くか」が決まる材料** であって、コード自体ではない。

実装に着手するか、計画書として書き起こすか、ユーザーが判断できる形で方針を提示する。**「とりあえず書いてみました」では役割を果たしていない**。

ユーザーが「方針 OK、実装して」と明示した時点で、初めてコード修正フェーズに移行する。
