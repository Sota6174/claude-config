# トークン効率の高い調査の進め方

調査でトークンを浪費する典型パターンを避け、必要十分な情報で判断する。

## 大原則

> **検索の前に「何を知りたいか」を1行で書く**

クエリが曖昧だと、ヒットの取捨選択で context を浪費する。具体的には：

- ❌ 「Next.js について調べる」
- ✅ 「Next.js 15 で App Router を採用するメリットと、Pages Router からの移行コストの相場」

「何を判断したいか」を1行に落とせない時点で、調査の準備ができていない。落とせるまで考える方が結果として早い。

---

## ツールごとの使い分け

### context7 MCP（最優先）

**用途：** 公式ドキュメントを構造化された形で取得

**手順：**
1. `mcp__plugin_context7_context7__resolve-library-id` でライブラリ ID を確定
2. `mcp__plugin_context7_context7__query-docs` で **topic を絞って** 取得

**コツ：**
- 同一ライブラリで複数トピックを聞きたい場合、`topic` を変えて連続呼び出しではなく、**まとめて聞ける表現**にする
- 「とりあえず全部見せて」は禁止。`topic` を指定してドキュメント範囲を絞る
- バージョン指定が効くなら活用する（古いメジャーバージョンの API を取らないため）

### WebSearch

**用途：** 「比較情報」「現場の評判」「特定エラーの解決策」など、公式ドキュメントに書かれない情報

**コツ：**
- クエリに **年（例: 2026）** を含めると新しい情報が優先される
- 検索結果の **タイトル + スニペット** で取捨選択し、本文を取りに行くのは **2〜3件に絞る**
- 「<候補A> vs <候補B>」「<候補> alternatives」「<候補> deprecated」「<候補> production」のような比較系クエリが効果的

### WebFetch

**用途：** WebSearch でタイトル / スニペットから「これは読む価値ありそう」と判断したページの本文取得

**コツ：**
- 同じ URL に何度も WebFetch しない。**1回で取得し、抜粋して使い回す**
- 大きなページは「探したいキーワード」を `prompt` 引数で指定して、関連箇所だけ抜く
- 公開日を必ず確認する

### Agent / Explore（補助）

**用途：**
- 既存プロジェクト内のコードや設定ファイルを横断調査したい時 → Explore
- 複数の独立した調査を並列に走らせて context を節約したい時 → 並列 Agent

**コツ：**
- 並列スパウンは「独立した質問が複数ある」時のみ。依存関係があるならシリアル
- 自分の context に取り込みたくない探索（リポジトリ全体検索など）は Agent / Explore に任せて要約だけ受け取る

## やってはいけない調査パターン

### 1. 同じことを何度も検索する
最初に取得した情報を本文中で参照しながら蓄積していけば、後段で再検索は不要。**「あれもう一回検索しよう」と思った時点で、最初の調査記録が雑だった**と疑う。

### 2. 大きなドキュメントを丸ごと取得して読み直す
ページの一部だけ知りたいなら、`topic` / `prompt` で絞る。**全文取得は最終手段**。

### 3. 個人ブログを連鎖的にハシゴする
ランク C 情報は **裏取り目的** で当たり、それ自体を判断根拠にしない。1つ読んで2つ目を当たる前に、まず公式に戻る。

### 4. 検索結果に出てきたすべてのリンクをチェックする
WebSearch の結果から **2〜3件** に絞って WebFetch。それで足りなければクエリを変えて再検索。**「念のため全部見る」は無駄**。

### 5. 全候補を等しく深掘りする
候補が4個出たら、まず **3軸（例：機能適合度・学習コスト・運用コスト）を5分で評価**して、明らかに下位2個を切る。残り2個で深掘りする。

## 判断保留時の対応

調査しても確信度が中以下にしかならない場合：

1. **何が分からないかをユーザーに明示する**
2. ユーザーに追加判断材料の有無を聞く（実機で試したい、特定環境での挙動を確認したい等）
3. **小さく試す案を提案する**（例：「最小限のサンプルを <候補A> と <候補B> で書いて、1日触ってみてから決めるのはどうですか」）

「もう少し調べます」と粘り続けるより、不確実性をユーザーに渡して判断してもらう方が建設的なケースが多い。
