# 技術スタック比較表テンプレート

候補が3〜4個に絞れたら、次の表で評価する。**全項目を等しく深掘りしない** ── 明らかに脱落するものは早めに切る。

## 基本テンプレート

```markdown
| 評価軸 | 候補A | 候補B | 候補C |
|---|---|---|---|
| **機能適合度**（要件をどれだけカバー） | ○ / 〜 | △ / 〜 | × / 〜 |
| **学習コスト**（既存知識との親和性） | | | |
| **運用コスト**（ホスティング・ランタイム・依存） | | | |
| **コミュニティ活発度**（直近1年のリリース頻度） | | | |
| **ドキュメント品質** | | | |
| **ライセンス / 利用規約** | | | |
| **ユーザー固有制約への適合**（言語・環境・無料枠等） | | | |
| **将来性 / リスク**（Deprecation / 競合との競争） | | | |
```

評価は `○ / △ / ×` または `S / A / B / C` で。短いコメントを必ず添える。

## 軸の補足

### 機能適合度
- 要件の **マスト** をどれだけカバーするか、**できればほしい** をどれだけカバーするか、を分けて評価
- マストが満たせないなら即脱落

### 学習コスト
- ユーザーの現在の技術スタックとの距離（言語・パラダイム・周辺ツール）
- 公式ドキュメント / チュートリアルの品質も含める

### 運用コスト
- ホスティング費用（クラウド / セルフホスト / SaaS）
- ランタイム要件（メモリ / Node バージョン / OS 互換性）
- 依存パッケージの多さ・セキュリティ脆弱性履歴

### コミュニティ活発度
- 直近1年のリリース頻度（公式 Releases ページ）
- Issue / PR の応答速度（公式リポジトリ）
- Stack Overflow タグの質問数（活発さの目安）

### ドキュメント品質
- 公式ドキュメントの網羅性・更新頻度
- 多言語対応の有無
- example コードの実用性

### ライセンス / 利用規約
- OSS ライセンスの種類（MIT / Apache / GPL / 商用制限あり）
- SaaS の場合は料金体系・無料枠
- 商用利用時の制約

### ユーザー固有制約
- ユーザーの環境（Windows / Mac / Linux / モバイル）
- 言語制約（日本語 UI / 日本語ドキュメントの有無）
- 予算（無料枠 / 個人利用範囲）

### 将来性 / リスク
- メンテナの活動状況
- 競合との関係（明らかに失速傾向か）
- メジャー破壊的変更の予告

## 推奨セクションのテンプレート

比較表の下に、必ず次を書く：

```markdown
## 推奨：<候補名>

**理由（上位3つ）**
1. <ユーザー制約 X> に対して最も適合（一次情報：<URL or Issue 番号>）
2. <別の利点>
3. <別の利点>

**他案を選ばない理由**
- <候補B>：<具体的な不適合点>
- <候補C>：<具体的なリスク>

**残るリスク・注意点**
- <未解決の懸念>
- <将来見直す可能性のあるポイント>

**前提となる確信度**
- 高 / 中 / 低 — <一次情報での裏取り状況>
```

選択肢を並べて終わりにしない。**推奨1案を必ず出す**。

---

## 比較表を作らなくていい場面

- 候補が1つしか妥当でない時（その場合は理由を3行で）
- ユーザーが既に決めていて、確認だけ求めている時
- 軽微な選択（例：ロガーの色付け方）

判断に迷うなら作る側に倒す。比較表を作る方が、後から見直す時にも有用。
