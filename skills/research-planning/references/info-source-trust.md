# 情報源の信頼性ランキング

技術調査で最も避けたいのは、**古い情報 / 誤った情報 / バイアスのある情報** に基づいて方針を決めること。後の修正コストは桁違いに大きい。

そこで、情報源は次の優先順位で取得・引用する。

## ランク S：一次情報（最優先）

| ソース | 強み | 弱み | 取得方法 |
|---|---|---|---|
| 公式ドキュメント | 正確性・最新性が原則担保 | 概念説明が薄いことがある | **context7 MCP**（`resolve-library-id` → `query-docs`）を最優先 |
| 公式 GitHub リポジトリ | バージョン・廃止予定・既知バグの最新状況 | 大量で読みづらい | README / CHANGELOG / Issues / Releases を WebFetch |
| 公式ブログ / リリースノート | バージョン間移行情報・将来計画 | 古い記事は誤情報源になる | 公開日を必ず確認 |
| 公式コミッターの発信 | 設計意図・推奨パターンの解説 | 個人見解が混じる | 肩書きを確認 |

**ルール：** 採用判断の決め手になる情報は、必ずランク S で裏取りする。

## ランク A：準一次情報

| ソース | 用途 |
|---|---|
| 公式の RFC / Spec ドキュメント | プロトコル・標準への準拠確認 |
| 有名 OSS の公式 example リポジトリ | 推奨パターンの確認 |
| カンファレンス公式講演動画 / スライド | 設計思想の理解 |

## ランク B：高品質な二次情報（比較情報の補強として）

| ソース | 用途 | 注意 |
|---|---|---|
| 大手技術メディア（MDN, web.dev, AWS Docs, Microsoft Learn 等） | 横断的な比較・解説 | 個別ライブラリへの言及は鮮度を確認 |
| StackOverflow（高評価回答） | 具体的な落とし穴・解決策 | 古い回答が上位に残りやすい、**回答日を確認** |
| Hacker News（コメント） | 各候補の批判的見解 | ノイズが多い、複数コメントで傾向を掴む |
| Reddit `/r/programming` 系サブ | 現場感のあるレビュー | バイアスあり、複数スレッドで傾向確認 |

## ランク C：参考情報（裏取り必須）

| ソース | 注意 |
|---|---|
| 個人技術ブログ（Qiita / Zenn / dev.to / Medium） | 鮮度・著者の専門性を必ず確認。**裏取りなしでの引用は禁止** |
| ChatGPT / Claude などの生成回答（過去ログ） | 古いトレーニングデータ起因の誤情報リスク |

## ランク D：使わない

- ソースを示さない記事
- 鮮度不明（公開日記載なし）の技術記事
- マーケティング色の強いベンダー比較サイト

---

## 情報を採用する3つのテスト

ある情報を採用する前に、次の3点を確認する：

1. **新しさ** — 公開・更新日が「現在のメジャーバージョン」と整合するか？（半年〜1年以上前の情報は要再確認）
2. **独立性** — 同じ結論を**別の独立したソース**でも確認できたか？（同じソースの引用ばかりではダメ）
3. **一次接続** — 結論の根拠が一次情報（公式ドキュメント・公式 Issue 等）に辿れるか？

3つ全てを満たさない情報は「参考情報」止まりにして、判断の決め手にはしない。

---

## 食い違いに出会ったら

複数ソースで結論が食い違うことはよくある。その時は：

1. **新しい側 / 公式に近い側を優先**する
2. **食い違いの存在自体をユーザーに明示**する（隠さない）
3. 重大な判断材料なら、公式 Issue / Discussion を直接当たって裏取りする

「公式ドキュメントには A と書いてあるが、半年前の人気記事は B と書いている」のような状況は、しばしばライブラリの仕様変更が原因。鮮度を疑うこと。
