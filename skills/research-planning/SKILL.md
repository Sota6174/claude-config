# research-planning

「アイデア」と「動くコード」の間にある"設計判断"を、**情報の信頼性最優先 × トークン効率**で支える調査スキル。

要件定義・構想段階での技術判断は、後の修正コストに直結する。だからこそ「最新かつ一次情報に近いソース」「複数情報源の突き合わせ」「ユーザー固有制約への適合度」の3点で根拠を固めた上で提案する。

---

## いつ使うか（トリガーの再確認）

次のような発言があったら発動する：

- 「〜を作りたい / 始めたい」「〜の作り方を考えたい」
- 「何の言語/フレームワーク/ライブラリ/サービスを使うべき？」「比較してほしい」
- 「アーキテクチャを検討したい」「構成を相談したい」
- 「最新のベストプラクティスは？」（特定コードへの修正ではなく方針相談）
- 「○○と××どちらがいい？」

**逆に使わない場面：** 既存コードの修正方針、特定ライブラリのAPI使い方の確認、バグ修正 → これらは [[research-implementation]] の領分。

---

## 調査フロー

ステップを飛ばさず順に実行する。なぜなら、信頼性の確保とトークン浪費の回避は「順序を守ること」でしか両立しないから。

### Step 1. 要件と制約を言語化する（コード/検索を始める前）

ユーザーの依頼から次を抽出し、不足は **AskUserQuestion で1ラウンドだけ確認する**：

- **目的**：何を実現したい / 解決したい問題
- **入出力**：扱うデータ・想定ユーザー・想定規模
- **制約**：使える言語/環境（Windows/Mac/サーバー/ブラウザ）、予算、コスト感、運用形態（個人/業務）、既存資産との連携
- **非機能要件**：パフォーマンス、可用性、学習コスト、保守性のうち重視するもの
- **除外条件**：「これは使いたくない」「これは難しい」

抽出できたら **1〜3行のサマリでユーザーに復唱**し、認識合わせをしてから調査に入る。これを省くと、調査結果が外れていた時に全部やり直しになる。

### Step 2. 候補の当たりをつける（既存知識で粗く）

調査ツールを呼ぶ前に、**現時点で知っている候補を2〜4個列挙する**。これは検索クエリを絞り、context7 で resolve するライブラリ ID を決めるための準備。当たりが付かない領域なら Step 3 の WebSearch から入る。

### Step 3. 一次情報を取りに行く（信頼性ランクに従う）

`references/info-source-trust.md` の信頼性ランキングに従い、**ランクの高いソースから取得**する。

- **最優先**：`mcp__plugin_context7_context7__resolve-library-id` → `query-docs` で公式ドキュメントを取得
- **次点**：公式 GitHub の README / docs / CHANGELOG（バージョン固有の挙動・廃止予定の確認）
- **補助**：公式メンテナの発信（公式ブログ、コミッターの記事）
- **比較情報の補強として**：WebSearch で「<候補名> alternatives 2026」「<候補名> vs <他候補名>」を引く（**StackOverflow / Reddit / Hacker News / 知名度の高い技術メディアに限る**、個人ブログは裏取り必須）

> **Claude/Anthropic 系プロジェクト（Agent SDK / MCP / Claude API 利用 / LLM アプリ）を選定する場合は、必ず先に `claude-api` skill を呼ぶ**。モデル ID・価格・パラメータの幻覚を防げる。詳細は [[claude-api]]。

**重要な原則**：
- 古い情報は害になる。検索結果に出た記事は **公開年・更新日を必ず確認**し、メジャーバージョン更新後の情報か検証する
- 1つのソースで結論を出さない。**最低2つの独立した情報源**で同じ結論が得られたら採用する
- 情報の食い違いがあったら、**新しい / 公式に近い側を優先**し、食い違いの存在自体をユーザーに報告する

### Step 4. ユーザー制約への適合度で評価する

`references/comparison-template.md` の比較表テンプレートに沿って、候補を以下の軸で評価する：

- 機能適合度（要件をどの程度カバーするか）
- 学習コスト・既存知識との親和性
- 運用コスト（ホスティング / ランタイム / 依存関係の管理）
- コミュニティの活発さ（直近1年のリリース頻度、未対応 Issue 数の傾向）
- ライセンス・利用規約上の制約
- ユーザー固有制約（言語、環境、無料枠の有無など）への適合

各軸を「○/△/×」または短いコメントで埋める。**全候補を等しく深掘りしない** ── 明らかに脱落するものは早めに切る。

### Step 5. 推奨案を提示する

比較表に加えて、**1案を推奨し、その根拠と他案を選ばない理由**を明示する。形式は以下：

```markdown
## 推奨：<候補名>

**理由（上位3つ）**
1. 〜（ユーザーの制約 X に対して最も適合）
2. 〜（一次情報での挙動が安定）
3. 〜（学習コストとのバランス）

**他案を選ばない理由**
- <候補B>：〜の制約が合わない
- <候補C>：〜のリスクがある（公式 Issue ##### で未解決）

**残るリスク・注意点**
- 〜
```

「お任せ」「とりあえず」の依頼でも、必ず推奨を出す（選択肢羅列で終わらせない）。ただし**残るリスクや確信度の低い部分**は明示し、ユーザーが判断材料を持てるようにする。

### Step 6. 採用するなら BP md 化を提案する

推奨スタックが固まり、ユーザーが採用方針を示したら、**頻繁に参照することが予想される技術**については BP md 化を提案する。提案先は基本 `<project>/docs/best-practices/<tech>.md`。

提案文の例：
> このプロジェクトでは `<tech>` を中心に使っていく方針なので、公式ドキュメントの該当セクション（バージョン / API / 推奨パターン）を `docs/best-practices/<tech>.md` にまとめておくと、後の実装フェーズで毎回ドキュメントを取り直さずに済みます。今ここで作っておきますか？

実際の生成は [[research-implementation]] の `bp-md-template.md` を使う想定。このスキル単体では「提案まで」で止めて良い。

### Step 7. 次フェーズへの橋渡しを示す

採用方針が決まったら、ユーザーが次に何をすべきか明示する:

- **新規プロジェクト立ち上げの場合** → `init` skill で CLAUDE.md を生成する流れを提案する（プロジェクト規約・採用スタック・build/test コマンドの初期化）。詳細は [[init]]
- **既存コードへの実装フェーズに入る場合** → [[research-implementation]] にバトンを渡す（既存把握 → 公式 BP 照合 → 差分整理）

## トークン効率の原則（毎ステップで意識）

- **検索/取得の前に、何を知りたいかを1行で言語化する。** クエリが曖昧だと無駄なヒットで context を浪費する
- **context7 は同一ライブラリで複数トピックを聞かれそうなら、resolve-library-id 後に query-docs を1回でまとめる**。再呼び出しは最小限に
- **WebSearch の結果はタイトルとスニペットで取捨選択**し、本文を取りに行くのは2〜3件に絞る
- **同じことを2回検索しない**。出てきた情報は本文中で参照しつつ蓄積し、後段の Step で再利用する
- **巨大な公式ドキュメント全部読みは避ける**。トピックを絞って取得する（例：`mcp__plugin_context7_context7__query-docs` の `topic` 引数を活用）

## アウトプット様式

最終応答は次の構造で出す（短く済む依頼なら一部省略可）：

```markdown
## 要件サマリ（私の理解）
- 目的：
- 主要制約：
- 重視ポイント：

## 比較した候補
（比較表 — references/comparison-template.md を参照）

## 推奨：<候補名>

**理由**
1.
2.
3.

**他案を選ばない理由**
-

**残るリスク・注意点**
-

## 次のアクション
- [ ] BP md 化を提案するか（採用方針が固まれば）
- [ ] research-implementation スキルで実装段階に進むか
```

## 最後に：選択肢羅列で逃げない

ユーザーは「複数選択肢を並べて選ばせる」ことを求めていない。**推奨を1つ出し、その理由を一次情報で裏付ける**こと。複数案を出すのは、ユーザーが「比較したい」と明示した場合か、決定的な判断材料が足りない時に限る。

「決め切るための調査」をするのがこのスキルの役割。
